# Demo 03 — Copilot Code Review
**Section:** 02 | **Presenter:** Chikamso | **Duration:** ~5 min | **Format:** Browser (PR review)

---

## What This Demo Shows

Requesting GitHub Copilot as a code reviewer on the PR generated by the coding agent (ROUTE-001). Walking through the Copilot review comments to show what it catches, what it explains, and how it improves the PR before human reviewers see it.

---

## Pre-Demo Setup

- The PR from Demo 02 (coding agent) must be open
- If running ahead of time: request Copilot as reviewer before the session starts — the review will be ready to show without waiting
- If running live: the review typically completes within 60 seconds of requesting

---

## Demo Script

### Step 1 — Request Copilot as Reviewer (1 min)

**On the open ROUTE-001 PR:**

**Say:**
> "The coding agent opened the PR. Before I ping a human reviewer, let me add Copilot as a reviewer first — let it do the first pass."

**Do:**
- In the **Reviewers** section (right sidebar): click the gear
- Type `copilot` → select **GitHub Copilot**
- Click away to confirm

**Say:**
> "Copilot will review this PR in about 60 seconds. It reads the diff, checks it against the codebase context, and leaves inline comments — just like a human reviewer would."

*(If using pre-run review: "I've already requested it ahead of time — let's look at what it found.")*

---

### Step 2 — Walk Through the Review (3 min)

**Navigate to the Files changed tab of the PR.**

**Say:**
> "Here are the review comments."

**Walk through 2–3 specific comments. Look for these types — highlight whichever appear:**

#### Comment type 1 — Logic suggestion
Example: Copilot flags that `onTimePercentage` is calculated before checking if `routeData` is empty, which would produce `NaN`.

**Say:**
> "This is a logic issue — if `routeData` is empty, this calculation returns NaN and the panel renders 'NaN%'. A human reviewer might catch this in testing. Copilot caught it in the diff."

#### Comment type 2 — Accessibility / best practice
Example: Copilot suggests adding an `aria-label` to the summary panel for screen reader accessibility.

**Say:**
> "This is a best-practice comment — not a bug, but something that matters for production quality. Copilot knows accessibility standards and applies them proactively."

#### Comment type 3 — Security awareness
Example: If any string is being injected into the DOM using `.innerHTML` (even from the local `routeData`), Copilot flags it.

**Say:**
> "This is the kind of comment that prevents the security issue we'll see in Section 5. Copilot flags unsafe DOM manipulation even in what looks like internal data flow. We'll come back to this when we talk about GHAS Autofix."

---

### Step 3 — The Review Summary (1 min)

**Scroll to the top of the review (Copilot's summary comment):**

**Say:**
> "At the top, Copilot gives a summary of the whole review — what it checked, what it found, and an overall assessment. In this case: changes look correct, two suggestions to address before merging."

**Highlight:**
- The summary distinguishes between **blocking** (must fix) and **non-blocking** (suggestions) comments
- "A human reviewer can now focus on the architecture and business logic — Copilot has already handled the surface-level issues."

---

## Key Talking Points to Land

1. **First-pass review in 60 seconds** — Copilot reviews faster than any human; humans get a cleaner PR to review
2. **Line-level comments** — not just a summary; specific, actionable feedback on specific lines
3. **It reads both the diff AND the codebase context** — it knows what patterns already exist and flags deviations
4. **Non-blocking vs blocking** — Copilot distinguishes suggestions from must-fixes, same as a good human reviewer
5. **Security-aware by default** — even on "safe" internal code, it flags XSS-risk patterns
6. **Reduces review fatigue** — human reviewers spend less time on style and obvious bugs, more time on what matters

---

## What to Highlight vs Skip

| What to spend time on | What to skip |
|---|---|
| A specific logic bug comment (most impactful) | The full list of every comment |
| The security-related flag (connects to Section 5) | Style/formatting comments |
| The review summary at the top | GitHub UI navigation |
| "60 seconds vs hours of human queue time" | How to configure review settings |
